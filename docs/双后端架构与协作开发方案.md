# 双后端架构与协作开发方案（技术面试 / 非技术面试）

> 目标：在现有项目上拆分出两套可并行开发、可独立演进的后端逻辑，分别服务“技术面试”和“非技术面试”，并保持前端接入方式统一。

---

## 1. 背景与目标

当前项目后端逻辑主要围绕技术面试流程构建。随着玩法扩展，需要新增“非技术面试”模式（如行为面、沟通面、管理面等），两种面试在以下方面差异明显：

- 题库类型与评分维度不同
- 追问策略与事件机制不同
- 提示策略与通过阈值不同
- 结果总结与反馈风格不同

为了避免逻辑强耦合、多人开发冲突、测试互相影响，建议拆分为**两个后端子项目**，并抽取共享能力层。

## 3. 请求流设计（前端无感切换）

前端在创建会话时携带字段：

- `interviewType: "tech" | "non_tech"`

后端路由建议统一前缀：

- `POST /api/session/start`
- `POST /api/session/answer`
- `GET  /api/session/{id}/result`

内部转发策略：

1. 若 `interviewType = tech` → 调用 `tech_interview_backend`
2. 若 `interviewType = non_tech` → 调用 `non_tech_interview_backend`

这样前端无需维护两套 API，仅在参数层切换模式。

## 4. 两套后端的职责边界

### 4.1 技术面试后端

适合保留你们当前逻辑：

- 技术题问答、追问深挖
- 正误判定与提示机制
- 技术事件（架构质疑、方案压测、代码解释）
- 技术能力维度评分

### 4.2 非技术面试后端

建议新增以下能力：

- STAR 行为题判定（场景、任务、行动、结果完整度）
- 沟通表达评分（结构化表达、简洁度、冲突处理）
- 价值观/文化匹配维度
- 情景题事件（跨部门冲突、优先级博弈、压力沟通）
- 非技术提示机制（引导补充而非直接给答案）

## 5. 并行开发分工建议（小组协作）

### A 组：技术面试后端（稳定 + 优化）

- 负责目录：`backends/tech_interview_backend/`
- 目标：迁移现有引擎 + 稳定 API + 回归测试

### B 组：非技术面试后端（新能力）

- 负责目录：`backends/non_tech_interview_backend/`
- 目标：完成非技术引擎、题库、人设与评分体系

### C 组：共享层与前端接入

- 负责目录：`backends/shared/` + `web/`
- 目标：统一协议、会话模型、错误码、日志与埋点

---

## 6. 接口契约（建议先冻结）

建议先定义并冻结 `session` 统一响应结构，避免前后端返工：

- `sessionId`
- `interviewType`
- `round`
- `question`
- `options`（若有）
- `timer`
- `scoreDelta`
- `stressDelta`
- `gameOver`
- `result`（仅结束时返回）

> 两个后端必须保证字段兼容；即使内部算法不同，返回格式保持一致。

## 7. 迁移步骤（建议顺序）

1. ~~复制顶层 `backend/` 到 `backends/tech_interview_backend/`~~（已完成；重复目录已删除，技术后端仅此一处源码）。
2. 抽取通用模块到 `backends/shared/`（AI 客户端、通用工具、公共常量）。
3. 创建 `backends/non_tech_interview_backend/` 最小可运行版本（MVP）。
4. 在 `server.py` 增加 `interviewType` 分流逻辑。
5. 前端在开局页增加“技术/非技术面试”入口选择。
6. 增加基础自动化测试，至少覆盖 start/answer/result 三条主链路。

## 8. 风险与规避

- **风险 1：共享层持续膨胀**
  - 规避：仅抽“稳定且通用”的能力，业务逻辑禁止放 shared。

- **风险 2：两套 API 逐步漂移**
  - 规避：维护单一接口契约文档 + 基于契约的测试。

- **风险 3：题库与评分规则耦合过深**
  - 规避：题库配置化，评分器模块化（策略模式）。

---

## 9. 里程碑建议

- **M1（1~2 天）**：目录拆分 + 技术面后端迁移可跑通
- **M2（2~3 天）**：非技术面后端 MVP（3 名面试官 + 基础评分）
- **M3（1~2 天）**：统一接口 + 前端联调 + 回归测试
- **M4（冲刺）**：体验优化、事件丰富、文案润色、发布演示版

---

## 10. 结论

本方案采用“**双后端 + 共享层 + 统一接口契约**”的方式，既能保证你们现有技术面逻辑稳定迭代，也能让非技术面试逻辑独立快速演进，适合小组并行开发与短周期比赛冲刺。
