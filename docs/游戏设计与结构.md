# 终面：AI面试官 · 项目说明文档

> 一份完整的玩法 + 设计 + 结构总结文档。对应代码版本：`backends/tech_interview_backend/` + `web/` + `server.py`。

---

## 一、产品定位

| 项目 | 内容 |
| --- | --- |
| 名称 | `终面：AI面试官`（ShowMeTheOffer · AI Interview Survival） |
| 形式 | 基于浏览器运行的 **AI 驱动动态面试生存游戏** |
| 核心循环 | 简历 → 面试邀请 → 仿腾讯会议面试间 → Offer / 未录用 |
| 运行模式 | **Mock 模式**（无需模型，本地规则跑通）+ **LLM 模式**（接 OpenAI 兼容接口，出题/判题/提示/总结/Offer 由大模型生成） |
| 技术栈 | 纯原生：Python 标准库 HTTP 服务 + 原生 HTML/CSS/JS 前端，无第三方依赖 |

设计目标：让玩家在一局（通常 3–6 轮）"真实仿真的面试"中体会 **答题、压力、深挖、随机事件** 交织的戏剧感，而不是一般的问答测验。

## 二、核心玩法

### 2.1 总体流程（4 步）

1. **Step 1 · 配置简历**
   - 选择岗位（`AI 游戏策划 / 前端交互 / AI 应用工程师 / AI 产品策划`）
   - 选择难度（`easy / normal / hard / expert / master`）
   - 简历模式：**自定义粘贴** 或 **AI 生成一份带风险点的简历**
2. **Step 2 · 收取面试邀请**
   - 系统分析简历（命中关键词、风险点、追问焦点）
   - 从面试官池中 **随机抽 3 位** 给玩家发出邀请卡，玩家任选其一
3. **Step 3 · 仿腾讯会议面试间**
   - 1 位主面试官 + 玩家自我格 + 2 个灰态旁听格（HR / 录制 AI）
   - 实时字幕区 + 倒计时 + 作答输入区 + 深挖指示灯 + 事件弹窗
4. **Step 4 · 结果结算**
   - 达标 → Offer 函（公司名、岗位、薪资、正文、签名）
   - 未达标 → Reject 卡片
   - 附：六维雷达 / 高光题 / 翻车题 / 轮次得分 / 分享文案

### 2.2 一轮面试内部规则

每一轮都围绕 **一道题** 展开，满分 `100`，轮分由以下事件动态扣减：

```
开始轮次：roundScore = 100
  └─ 答题（有倒计时）
      ├─ correct  → 掷骰深挖（drillDepth < 3）
      │    ├─ 命中 → 追问（再答一次，同轮）
      │    └─ 未中 → 本轮结束
      ├─ partial  → 扣 12 分 + 掷骰深挖
      └─ wrong    → 扣 28 分 + 掷骰提示（hintsUsed < 3）
           ├─ 命中 → 再给一次提示并重答（扣 8 分/次）
           └─ 未中 → 本轮结束
  └─ timeout → 本轮分数压到 ≤10，直接结束
```

关键常量定义在 `backends/tech_interview_backend/engine.py`：

```python
ROUND_BASE_SCORE = 100
DRILL_PENALTY_PARTIAL = 12
DRILL_PENALTY_WRONG = 28
HINT_PENALTY = 8
TIMEOUT_SCORE = 10
```

### 2.3 通关判定

- **轮数**：`random.randint(min_rounds, max_rounds)`，由面试官配置（普通 3–5，黑盒 AI 4–6）。
- **轮分**：每轮结算时再乘难度系数（`score_multiplier` 0.85–1.3）。
- **综合分**：所有轮分平均。
- **过线**：综合分 ≥ 面试官的 `pass_score`（68 / 72 / 75 / 80 各不相同）。
- **特例**：部分随机事件可以 **直接 `ends=true`** 导致面试立刻结束（例：黑盒 AI 文字题超时会被判"样本不可验证"，当场终止）。
- 六维评分 (`roleFit / logic / depth / consistency / composure / adaptability`) 仅作复盘展示，不影响通关。

### 2.4 随机事件系统

- 每位面试官自带一组 `random_events`，按 **轮内 / 轮间** 两个触发点掷骰。
- 事件类型：
  - `impact=false` 的 **旁白事件**（不交互，如"手机震了一下"）
  - `impact=true` 的 **选择题**（3–4 个选项 + 超时惩罚）
  - `impact=true` 的 **文字输入题**（关键词命中判分 + 超时惩罚）
- 事件一局内每个最多触发一次（用 `firedEvents` 集合记录）。
- 事件会影响：本轮分 / 压力值 / 临场反应维度，部分直接 `ends` 终结面试。

## 三、4 位面试官（可扩展）

| ID | 人设 | 通过线 | 深挖概率 | 提示概率 | 特色 |
| --- | --- | --- | --- | --- | --- |
| `cold-judge` 冷面审判官 | 强压迫技术总监 | 75 | 0.85 / 0.55 / 0.3 | 0.45 | 事件：VP 突然进会议 / 简历质疑 |
| `spark-hr` 元气 HR | 看似友好、温柔拆解 | 68 | 0.7 / 0.45 / 0.2 | 0.75 | 事件：期望薪资试探 / Offer 二选一 |
| `tactical-lead` 战术分析师 | 方法论派 Tech Lead | 72 | 0.75 / 0.6 / 0.35 | 0.5 | 事件：白板画图 / 24 小时压线 |
| `blackbox-ai` 黑盒 AI 观察者 | 机械冷幽默的 AI 评估系统 v3.2 | 80 | 0.9 / 0.7 / 0.4 | 0.35 | 事件：置信度审计 / 样本可验证性 |

每位面试官在自己的 `.py` 文件里声明：**人设、开场白、邀请函、通过线、题库、深挖/提示概率、作答时长、随机事件表**，完全自包含。

**扩展方式**（零侵入）：在 `backends/tech_interview_backend/interviewers/` 下新建 `xxx.py`，导出 `INTERVIEWER = {...}`，框架在启动时会由 `pkgutil.iter_modules` 自动发现并注册，无需改动任何其它文件。

## 四、难度系统

```python
DIFFICULTIES = {
    "easy":   {..., "score_multiplier": 0.85, "starting_stress": 18, "max_turns": 4},
    "normal": {..., "score_multiplier": 1.0,  "starting_stress": 24, "max_turns": 4},
    "hard":   {..., "score_multiplier": 1.1,  "starting_stress": 32, "max_turns": 5},
    "expert": {..., "score_multiplier": 1.2,  "starting_stress": 40, "max_turns": 5},
    "master": {..., "score_multiplier": 1.3,  "starting_stress": 48, "max_turns": 6},
}
```

难度会联动：

1. **轮数上限**（进一步放大到 6 轮）
2. **起始压力值**（HUD 的压力条初始高度）
3. **轮分倍率**（最终 `roundScore` 再乘一次）
4. **AI 生成简历的风险点等级**（`low → extreme`，越高越容易被追问露馅）

### 5.3 状态机

核心阶段：`awaiting_answer → judging → (drill | hint | round_end) → event? → next_round → final`

```
start_session()
    └─> opening → awaiting_answer (R1)

submit_answer():
    judging → (correct | partial | wrong)
        correct / partial:
            roll drill_probability[drillDepth]
                yes & depth < 3  -> drillDepth++，awaiting_answer (同轮)
                no               -> round_end
        wrong:
            roll hint_probability & hintsUsed < 3
                yes              -> hint，awaiting_answer (同题)
                no               -> round_end (本轮折扣)

submit_timeout():
    记为本题 0 分，round_end (直接结束本轮)

round_end():
    roll inter_round_event
        无影响        -> 写 transcript，下一轮或 final
        有影响        -> 返回 phase=event，等 /session/event
    检查 roundIndex >= totalRounds -> final

submit_event():
    resolve → 加减分 → 如 ends=True -> final，
                        否则按原触发位置继续
```

### 5.4 会话模型（关键字段）

服务端用一个内存字典 `sessions[sessionId]` 维护每一局：

- 身份：`id / createdAt / role / interviewer / difficulty / themeKeyword`
- 简历：`resumeMode / resumeText / analysis`
- 状态机：`currentPhase / roundIndex / totalRounds / drillDepth / hintsUsed`
- 分数：`roundScore / roundScores[] / sessionStress / dimensions{6维}`
- 对话：`transcript[] / questionHistory[] / focusPoints / usedFocusPoints`
- 事件：`firedEvents(set) / pendingEvent / pendingEventTrigger`
- 结算：`ended / finalReport`

## 六、LLM / Mock 双轨设计

最值得强调的设计决策：**所有 AI 能力都可降级**。

- `ai_client.configured` 只有在检测到 `OPENAI_API_KEY + OPENAI_MODEL` 且 `AI_GAME_FORCE_MOCK != 1` 时返回 true。
- 每次调用 LLM 都包在 `try / except AIClientError` 里，一旦出错或返回非法 JSON，都会 **静默回退到 `_local_*` 方法**：
  - 开场 → `_local_opening_question`
  - 出题（轮次/深挖） → `_local_drill_question`
  - 判题 → `_local_judge_answer`（基于关键词、数字、长度、模糊词等做规则加减分）
  - 提示 → `_local_hint`
  - 总结 → `_local_final_report`
  - Offer → `_local_offer_letter`

好处：

1. 比赛/演示时不依赖云端也能跑
2. 新开发者 `python server.py` 即跑通
3. 接入新模型时，Prompt 已经写成强制 JSON 输出（见 `prompts.py`），出错也不会炸

### Prompt 清单（`backends/tech_interview_backend/prompts.py`）

| Builder | 用途 | 温度 | 必备字段 |
| --- | --- | --- | --- |
| `build_opening_prompts` | 开场白 + 首问 | 0.9 | `openingLine / firstQuestion / focusPoints` |
| `build_question_prompts` | 轮次出题 / 深挖追问 | 0.85 | `question / focusHint` |
| `build_judge_prompts` | 判题 | 0.55 | `verdict / feedback / scoreDelta / dimensionDelta / flags` |
| `build_hint_prompts` | 提示生成 | 0.7 | `hint / rephrased?` |
| `build_summary_prompts` | 结算报告 | 0.8 | `verdict / verdictLabel / summary / interviewerQuote / highlight / flop / tips / shareLines` |
| `build_offer_prompts` | Offer 邮件 | 0.9 | `company / position / salaryRange / startDate / signature / body` |

## 七、前端关键体验设计（`web/`）

- **仿腾讯会议四宫格**：主面试官带"说话时高亮圈"、玩家格、HR 旁听、AI 录制。
- **HUD 四项**：综合分 / 本轮分 / 通过线 / 压力条（带小仪表）。
- **倒计时条**：答题条 + 事件条两条独立；超时自动调用 `/api/session/timeout`。
- **深挖指示灯**：三个圆点直观标记当前深挖层数。
- **事件弹窗**：根据 `interaction.type` 动态切换选择题 / 文字题 UI，带独立倒计时。
- **Transcript**：按 `speaker` (interviewer / question / candidate / feedback / system) 分类渲染，形成可回放的对话流。
- **结果页**：Offer / Reject 双形态 + 六维雷达 + 高光 / 翻车 + 分享文案。

## 九、一句话总结

> **《终面：AI面试官》** 是一个"轮次状态机 + 随机事件引擎 + 双轨 AI 调用层 + 仿腾讯会议前端"拼成的网页游戏：玩家用简历换入场券，用一次次回答对抗四位风格迥异的 AI 面试官，扛过 **深挖、提示、超时、随机事件** 的组合拳，争取一封 **动态生成的 Offer**。
